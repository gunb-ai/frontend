Showcase · backend dev

Every feature, on one CRUD service.

The EE-flavored showcase walks a low-pass filter. This one walks the same seven features on a stack a backend dev lives in: a Todo CRUD service. One .dag file declares the data, the service, the store, and the UI. Every artifact — Rust handler, TypeScript client, SQL migration, OpenAPI spec, docs, CI lens receipts — is a projection of that one graph.

One source of truth. Five emitted languages. Zero drift.

Step 1 · the graph

42-line .dag. The whole service.

The backend dev writes one file. It declares the data shape, the service surface (HTTP routes, methods, request/response types), the store (table schema + indexes), and a small UI. No hand-written migrations, no hand-written client. Every consumer of Todo stays in sync structurally — or the build fails closed before anything ships.

todo_app.dag Building next · one source of truth 
module examples.todo_app

import v4.std.http   { HttpGet, HttpPost, HttpPatch, HttpDelete }
import v4.std.time   { Timestamp }
import v4.std.uuid   { Uuid }

app TodoApp {
  data Todo {
    id:         Uuid
    list_id:    Uuid
    title:      String
    completed:  Bool
    created_at: Timestamp
  }

  type NewTodo  { list_id: Uuid, title: String }
  type Toggle   { completed: Bool }

  service TodoApi {
    get_by_id:    HttpGet</todos/:id>                -> Todo
    list_in_list: HttpGet</todos?list_id=:list_id>   -> List<Todo>
    list_all:     HttpGet</todos>                    -> List<Todo>
    list_recent:  HttpGet</todos?sort=created_at>    -> List<Todo>
    duplicates:   HttpGet</todos/duplicates>         -> List<TodoPair>
    create:       HttpPost</todos, NewTodo>          -> Todo
    toggle:       HttpPatch</todos/:id, Toggle>      -> Todo
    delete:       HttpDelete</todos/:id>             -> Unit

    spec latency_budget: each handler < 50ms p95
  }

  store TodoDb {
    table todos: Todo
      primary id
      index   (list_id, completed)
      index   (created_at desc)
  }

  ui TodoList {
    source:    TodoApi.list_in_list
    render:    TodoCard
    on_toggle: TodoApi.toggle
  }
}

Step 3 · cost lens per handler

Every handler gets a structural cost class.

The complexity analyzer folds across each handler's body and the store's indexes. Pick a handler to see its derived cost class, the resulting DB plan, and whether the latency-budget spec lens passes. This is what the analyzer reports today on weather.dag (playground) — pointed at a service surface.

fn get_by_id(id: Uuid) -> Todo {
  todos
    |> by_primary(id: id)
    |> first()
}

Hash/B-tree lookup over primary key is constant by construction. The analyzer folds the call chain (by_primary → first) and sees no dependence on table size n.

Same algebra as the complexity playground — but driven by the service-surface structure (HTTP route + store query + index declaration) rather than function pipelines on lists. The five handlers above cover the cost classes the lens can derive today.

Step 5 · lens gate

Latency budget enforced before merge.

The service declares a spec: each handler < 50ms p95. The cost lens derives a symbolic complexity per handler. The latency-budget lens combines that with table-size assumptions from the store and decides whether each handler fits — and fails closed on anything that doesn't, before the artifact is emitted. No more "we'll catch the regression in load testing."

latency_budget_lens · 7 / 8 handlers

PASS

get_by_id, list_in_list, list_recent, create, toggle, delete
all derive ≤ O(n log n) on the indexed columns; modeled p95 ≤ 50ms

latency_budget_lens · duplicates handler

FAIL · O(n²)

duplicates · pairwise comparison without an index
derived bound: O(n²) on todos table size
modeled p95 at n=10k: 3.2 s · budget: 50ms
suggested: add index (title) where completed=false, or compute duplicates via a stream that visits each title once

Step 6 · the agent path

An agent fixes the slow handler. Same gates.

The slider in the EE showcase was a UI affordance over a typed Diff. Here, an agent proposes the same kind of diff: add an index, rewrite the offending body, re-emit. The lens is the same lens a human would run; the gate is the same gate.

agent.ts — fix duplicates handler via apply_diff Building next · v4 SDK 
import {
  ingestText, foldNode, affectedSet,
  applyDiff, emit, addStoreIndex, replaceHandlerBody,
  type CoreNode, type Diff, type Outcome,
} from "@gunb-ai/gunbc";

const root = await ingestText(
  await Deno.readTextFile("examples/todo_app.dag"),
  { inputLang: "dag" },
);

// 1 · Diff: add an index, rewrite the handler to use it
const diff: Diff = {
  edits: [
    addStoreIndex("TodoDb", "todos",
                  "(title) where completed = false"),
    replaceHandlerBody("TodoApi.duplicates",
      `todos |> by_title_index_where_active()
              |> group_by(t => t.title)
              |> filter(g => g.size > 1)
              |> map(pair_from_group)`),
  ],
};

// 2 · Candidate state: apply_diff is sequential + fails closed (P3)
const candidate: Outcome<CoreNode> = applyDiff(root, diff);
if (!candidate.ok) throw candidate.diagnostics;

// 3 · Affected: re-emit only the artifacts the index + body touched
const affected = affectedSet(candidate.value, {
  at: "Service.TodoApi.duplicates",
});
// → [ "Rust::duplicates handler", "TS::client", "SQL::migration",
//     "OpenAPI::operation", "Docs::duplicates", "Lens::latency_budget" ]

// 4 · Gate: re-run the lens — does the bound now fit the budget?
const gated = foldNode(candidate.value, latencyBudgetLens);
// → { duplicates: "PASS · O(n log n) · modeled p95 = 18ms" }

// 5 · Emit only the affected artifacts (others skip with receipt)
await emit(candidate.value, { target: "rust",       outDir: "./svc/src" });
await emit(candidate.value, { target: "typescript", outDir: "./ui/src" });
await emit(candidate.value, { target: "sql",        outDir: "./migrations" });

Same primitives. Same affected set. Same gates. The agent never freehand-wrote a migration; the typed graph holds the agent to the same lens the human path runs.
