daglang · gunbc compiler

One graph. Many targets.

Modern systems drift because the backend, frontend, schema, tests, docs, CI, and runtime artifacts all become separate sources of truth. daglang represents the system as a typed graph. gunbc analyzes the graph, checks structural properties, and emits artifacts from the same source.

The problem

Drift is the default.

Every system above a certain size carries the same tax. The cost shows up at code-review time, at test time, at incident time, and at refactor time — but the root is always the same shape.

Schema drift. A field changes in the backend; the client compiles fine, then breaks at runtime. Caught in production, not at build.
Documentation drift. A type changes; the docs stay stale until someone notices. Onboarding gets harder; trust in docs erodes.
Cost drift. A helper gets slower; downstream code silently crosses a complexity budget. Tail latency rises; nobody knows which commit did it.
Cross-domain drift. A circuit model, UI control, and simulation output describe the same thing — but no compiler sees them as one system. Engineers reconcile by hand. The reconciliation goes stale.

The proposal

Make the system a graph.

A source line is a view. Underneath it are typed graph nodes — fields, variants, call edges, match arms, effect boundaries, target-artifact projections. Because the compiler sees the graph, it can answer structural questions: what changed, what depends on it, what tests must run, what artifacts must update.

Step 1 Source line What the developer (or agent) writes — a function body, a type declaration, a config field.
Step 2 Graph node The compiler resolves the span to typed CoreNodes — addressable units of meaning.
Step 3 Lenses + checks Complexity, type-shape, exhaustiveness, effect, latency — each folds over the node graph.
Step 4 Artifacts Rust backend, TypeScript client, SQL migration, OpenAPI spec, generated tests, circuit netlist — projections of the same graph.

Available now

The supported slice today.

What you can run against the public compiler right now. Everything below is verified end-to-end on the documented daglang subset; the broader roadmap lives in /roadmap.

● gunbc compiler — compile + run, with --target rust verified by cargo check; Go and Python emit paths exist on the same graph and are stabilizing. Source: github.com/gunb-ai/daglang
● Documented daglang subset — types (struct, sum-of-products), pattern matching, pure functions, pipeline operators, std collections. Fails closed on unsupported constructs with named diagnostics.
● Structural complexity analyzer — function_classes attached to every analyzed function on every compile. Real benchmark in examples · complexity analyzer. O(1), O(n), O(n * n) derived from recursive structure — no annotations.
● Contract-drift / fail-closed examples — add a variant to a sum type, every match that doesn't cover it fails the build. See examples · contract drift.

Where this is going

Beyond programming languages.

A "target" doesn't have to be a language. Anywhere structure travels — circuits, schemas, protocols, workflows — drift between artifacts is the bug daglang is built to remove. Most of these are Planned per the roadmap above; the substrate types for SPICE and Verilog already exist in the v4 tree.
