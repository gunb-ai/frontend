primitives · 01

Materials

Five materials. Each has a reason. Moss is structural continuity, not "success." Clay is a boundary, not "danger." Neither is a UI status color — no traffic-light usage anywhere in the system.

Voidmaterial

The ground. Near-black, slightly warm. Product surface.

Stonematerial

Warm grey. Primary text on dark. Patient, recessive.

Warm Whitematerial

Off-white, warm cast. Light-mode surface.

Mossaccent

Structural continuity. Link, held relation. Not "success."

Clayaccent

Boundary. Diagnostic, invariant break. Not "danger."

lessons · 15

From primitive concept to composition of rules

Six lessons. The source is canonical — real .dag code, expandable per card. The visualization is downstream — the SVG shows what the code produces in the graph. SVG carries intuition; code carries truth. Each lesson builds on the previous one, ending where rules (constraints) start to compose.

movement 1 · atoms

A declaration introduces a node

Everything starts with a declaration. A type with three variants creates a node with three internal options. Two declarations referencing each other create an edge. The graph grows one declaration at a time.

01 Node vertex · variant type
examples/weather.dag
-- a primitive: declare a variant type
type Condition =
  | Sunny
  | Cloudy
  | Rainy

The declaration creates one node in the graph: Condition. Inside the node lives the variant structure — three options that any reader of Condition will have to handle. The graph knows about the node and its shape. It doesn't yet know what to do with it.

02 Edge directed reference
examples/weather.dag
-- relate two nodes by referencing them
record Observation =
  condition   : Condition
  temperature : Celsius

Observation introduces a new node and two edges — one to Condition, one to Celsius. The edges aren't decorations; they're structural dependencies. Anything that reads Observation must reach the two referenced nodes. The dependency is now part of the graph, not folklore.

movement 2 · composition

Folds read the structure; projections derive from it

Once the structure exists, the compiler can read it. A fold names a result for every variant; a projection maps that reading into a target language. The same graph becomes Rust, TypeScript, OpenAPI — every artifact stays consistent because every artifact comes from the same source.

03 Fold catamorphism
examples/weather.dag
-- name a result for every variant
let describe : Condition -> String =
  fold
    | Sunny  -> "clear"
    | Cloudy -> "overcast"
    | Rainy  -> "wet"

A fold is a seam through every variant. The line passes once over each case — Sunny, Cloudy, Rainy — and exits to a single result on the right. The result of the fold becomes a new node in the graph — describe — that other nodes can reference. Composition continues.

04 Projection structure-preserving map
examples/weather.dag
-- derive target artifacts from the same node
project Observation -> rust
project Observation -> typescript
project Observation -> openapi
project Observation -> tests

A projection is a structure-preserving translation. The same Observation node becomes a Rust struct, a TypeScript type, an OpenAPI schema, an exhaustive test set. Every artifact is consistent with every other artifact because every artifact reads the same source. This is where the system feels different from hand-written glue.

movement 3 · rules

Rules compose; the compiler refuses to guess

Rules are not annotations. They're compositions of fold + projection that the compiler can check. If a fold misses a variant, no artifact emits. If a service is projected, every node it reaches inherits the projection requirement. These behaviors don't have to be added — they fall out of structure being load-bearing.

05 Boundary fails closed
examples/weather.dag
let describe : Condition -> String =
  fold
    | Sunny  -> "clear"
    | Cloudy -> "overcast"
    -- Rainy missing

Invariant —
  match on Condition is not exhaustive
  missing: Rainy
  result:  no artifact emitted

The third variant has no handler. The seam holds in moss across Sunny and Cloudy, then breaks into clay where Rainy should have been — and stops at the clay wall. The compiler reaches the missing case at typecheck time, not at runtime. No Rust struct, no TypeScript type, no test is emitted past the wall: the compiler refuses to guess.

06 Composition of rules constraint propagation
examples/weather.dag
-- a rule that composes the previous lessons
service WeatherStation =
  read    : () -> Observation
  history : () -> List<Observation>

-- projecting the service projects every field
project WeatherStation -> rust
project WeatherStation -> openapi

-- the compiler now requires:
--   Observation projects to rust AND openapi
--   List<Observation> projects to rust AND openapi
--   every fold on Condition is exhaustive

Projecting a service quietly propagates a constraint over every node it reaches. The rule isn't written; it's structural. Anything reachable from the projected node has to project too. Add a target later (say, TypeScript), and the same closure decides whether the rule still holds or whether you've discovered an unmodeled corner. Rules don't accumulate as comments — they accumulate as graph edges.

Six lessons. Two atoms compose into a structure. Two operations read that structure into target artifacts. Two rules emerge — fail-closed, and constraint propagation — without anyone writing an extra annotation. The structure is the contract.

03 Idempotent upsert

Effects declared upsert compose safely — three resources here, three moss dots on the diagram (one per resource). Re-running the same upsert against the same desired state changes nothing, so retrying after partial failure doesn't duplicate infra.
