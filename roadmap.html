Roadmap

The roadmap is a graph.

Capabilities are nodes. Dependencies are edges. The compiler core is the root; everything else composes from it. Tier color = maturity; dashed purple edges = planned dependencies. The structure is the honesty mechanism — no percentages, no shipping dates we can't keep.

The plan: one graph, every target.

available now stabilizing building next planned
● Available now daglang subset · gunbc compiler (compile + run + --target rust/go/python/dag) · Rust emission verified end-to-end · complexity analyzer with function_classes on every compile · fold_node substrate primitive
◐ Stabilizing Go output · Python output · fail-closed diagnostics on unsupported paths · public examples surface
○ Building next TypeScript client · OpenAPI · SQL schema · lens-gated CI receipts · affected-set CI · programmatic graph editing (apply_diff) + candidate state · omni-ingestion (round-trip translate from any source) · node-level test generation
◇ Planned full-stack omni-emission (backend · client · schema · docs · tests · CI gates from one graph) · Verilog (Shape A language target) · SPICE (Shape B data format) · protocols · workflows · v4 self-host (compiler in daglang)

The graph above shows dependencies between capabilities; the text rows below it list specifics by tier. apply_diff sits in Building next because it composes from fold_node (shipped). Omni-ingestion is the inverse arrow of omni-emission, sharing the language-spec grammars in both directions. Planned-tier edges are dashed to mark them as design direction, not delivery promise.
