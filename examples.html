Real compiler output · algorithms benchmark

The output below is captured verbatim from a local gunbc dev build run against a seven-function benchmark designed to exercise every cost class the analyzer derives today. Nothing is hand-authored or annotated — the analyzer folds across each function's match + recursion structure and reports the resulting class. duplicate_count calls count_in inside its own recursion; the analyzer composes those two linear walks and lands on O(n * n) without any user hint.

algorithms.dag · 7 functions, no annotations Real source 
module examples.algorithms

fn double(n: Int) -> Int { n + n }

fn classify(n: Int) -> String {
  if n > 0 { "positive" }
  else { if n < 0 { "negative" } else { "zero" } }
}

fn sum_list(xs: List<Int>) -> Int {
  match xs {
    Empty => 0
    Cons { head: h, tail: t } => h + sum_list(xs: t)
  }
}

fn length_acc(xs: List<Int>, acc: Int) -> Int {
  match xs {
    Empty => acc
    Cons { head: _, tail: t } => length_acc(xs: t, acc: acc + 1)
  }
}

fn evens_doubled(xs: List<Int>) -> List<Int> {
  xs |> filter(x => x % 2 == 0) |> map(x => x * 2)
}

fn count_in(xs: List<Int>, target: Int) -> Int {
  match xs {
    Empty => 0
    Cons { head: h, tail: t } =>
      if h == target { 1 + count_in(xs: t, target: target) }
      else { count_in(xs: t, target: target) }
  }
}

fn duplicate_count(xs: List<Int>) -> Int {
  match xs {
    Empty => 0
    Cons { head: h, tail: t } =>
      count_in(xs: t, target: h) + duplicate_count(xs: t)
  }
}

Server loops have a defined bound too

Non-terminating loops aren't O(?). The substrate has a named Forever bound (the system-limit representable bound) and structured iteration via repeat(Forever, body), so the cost lens still derives a defined per-iteration class — never "I don't know." See v3-modeling-analysis.md §"Forever and Bounded Iteration".

request_loop.dag (illustrative)defined bound
fn run_server(state: ServerState) -> Forever {
  repeat(Forever, fn() {
    let req = accept_request()         // O(1) per-iter
    let resp = handle(req: req)        // O(1) per-iter
    send_response(resp: resp)
  })
}

Programmatic graph editing — agents are one consumer

Most coding agents operate on raw bytes — write a string, hope it parses. daglang exposes the program as a typed CoreNode graph. The primitives below are named directly from design-v4-compiler-homomorphism.md: ingest_text (peripheral shim), fold_node (catamorphism — every pass is data, not code), apply_diff (sequential edits with candidate-state gating, fails closed on path resolution), and affected_set (the query primitive that drives affected-set propagation).

Programmatic surface (designed in v4)

Reads are fold_node. Writes are apply_diff. CoreNode is the canonical input whether it came from text ingest, a programmatic builder, a query-driven rewrite, or a round-trip translate — so an agent's patch is held to the same lenses as a human edit.

gunbc · TypeScript SDK sketchBuilding next
import {
  ingestText, foldNode, affectedSet,
  applyDiff, emit, type CoreNode, type Diff, type Outcome,
} from "@gunb-ai/gunbc";

// 1 · Ingest: peripheral text→CoreNode shim (parse + normalize)
const root: CoreNode = await ingestText(
  await Deno.readTextFile("weather.dag"),
  { inputLang: "dag" },
);

// 2 · Read: every pass is a fold over the canonical Node
const complexity = foldNode(root, complexityAlgebra);
//   → { "fn:freezing_locations": "O(n)", "fn:lookupCustomer": "O(1)", … }

// 3 · Query: derive the affected_set from a Path
const affected = affectedSet(root, {
  at: "fn:lookupCustomer/body/find",
});
//   → ["fn:enrich", "fn:processOrders", "fn:checkout"]

// 4 · Edit: apply_diff folds substitutions through the candidate root.
//     Sequential — each Edit's Path resolves against the post-prior state.
//     Any unresolved Path = whole Diff fails closed (P3).
const diff: Diff = {
  edits: [{
    at: "fn:lookupCustomer/body/find",
    replacement: indexLookup({ index: "customers_by_id" }),
  }],
};

const candidate: Outcome<CoreNode> = applyDiff(root, diff);
if (!candidate.ok) throw candidate.diagnostics;

// 5 · Gate: re-run only the lenses whose scope intersects affected
const reChecked = foldNode(candidate.value, complexityAlgebra);
//   reChecked["fn:processOrders"] === "O(n log n)"

// 6 · Emit: language-spec realization from the green CoreNode
await emit(candidate.value, { target: "rust", outDir: "/tmp/weather-out" });

Composes with LawfulRewriteWitness for dry-run (lens identifies I/O, witness substitutes, eval against a mock runtime). Same surface a future query-driven rewrite or self-modification path uses — the compiler editing its own source goes through modeled apply_diff, not implicit file writes.

Beyond programming languages

A target doesn't have to be a language. Verilog is a Shape A language target — the compiler emits it through the same path as Rust, against the spec at v4/extdeps/languages/verilog.dag. SPICE is a Shape B data format — a flat component + connection graph emitted by user .dag programs folding over typed Component values, modeled at v4/extdeps/formats/spice.dag (T-4.10 omni-emission probe, modeled 2026-05-16). End-to-end gunbc compile --target spice / --target verilog is still Planned-tier; the substrate types below are real.

SPICE — analog circuits as Shape B data

Verilog — digital hardware as a Shape A language target

Verilog gets the same treatment as Rust: the compiler emits it directly against the language spec. The shape below uses the real VerilogModule / AlwaysBlock / NetAssignment types from verilog.dag — the same structural carriers that verilog_concurrency_anchor.dag uses to anchor T-4.9 (continuous-assign + posedge always-block concurrency).
